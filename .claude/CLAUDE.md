# プロジェクト固有の指示

## Gitコミットメッセージ

コミットメッセージは必ず日本語で記述すること。

### コミットメッセージの形式
- タイトル行：変更内容を簡潔に日本語で記述
- 空行
- 詳細：変更の詳細を箇条書きで日本語で記述
- 空行
- フッター：Claude Codeの署名

### 例
```
機能Xを追加

- コンポーネントYを作成
- 関数Zを実装
- テストを追加

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude <noreply@anthropic.com>
```

## コード編集後のエラーチェック

コードの編集や新規ファイルの作成を行った後は、必ず以下のエラーチェックを実施すること。

### 1. 診断エラーの確認
`mcp__ide__getDiagnostics`ツールを使用して、編集したファイルにeslintエラーやTypeScriptエラーが発生していないか確認する。

### 2. エラーが発生した場合
エラーが発見された場合は、そのエラーを解消するまで作業を完了としない。以下の手順でエラーを解消すること。

- エラーの内容を分析する
- 該当箇所を修正する
- 再度診断を実行し、エラーが解消されたことを確認する

### 3. 確認するエラーの種類
- eslintエラー
- TypeScriptの型エラー
- モジュール解決エラー
- その他のコンパイルエラー

### 4. 実施タイミング
- ファイルの編集後
- 新規ファイルの作成後
- 複数ファイルを編集した場合は、全ての編集完了後

この手順を省略せず、必ず実施すること。

## コーディング方針（Copilotレビュー観点）

コードを記述する際は、以下の6つの観点を常に適用すること。これによりCopilotからのレビュー指摘を最小限に抑える。

### 1. 型安全性
- 全ての関数で戻り値の型を明示する（例: `function foo(): string { ... }`）
- `any`型の使用を避け、適切な型を定義する
- ジェネリクスを適切に活用する
- 型推論に頼らず、明示的に型を書く

### 2. 可読性と保守性
- マジックナンバーやマジック文字列を避け、定数として名前付きで定義する
- 複雑な条件式は早期returnやガード節で簡潔にする
- 不要な中間変数を削減する
- 関数やコンポーネントは適切な粒度に分割する

### 3. アクセシビリティ
- 全てのインタラクティブ要素に`accessibilityRole`を設定する
- ボタンやリンクには`accessibilityLabel`を設定する
- エラーメッセージには`accessibilityLiveRegion="polite"`を設定する
- フォーム要素には適切なラベルを関連付ける

### 4. パフォーマンス
- useCallbackやuseMemoを適切に使用する
- 不要な再レンダリングを避ける
- リスト表示では適切なkeyを設定する
- 重い計算は必要な場合のみ実行する

### 5. エラーハンドリング
- エッジケース（空配列、undefined、null等）を考慮する
- 適切なデフォルト値を設定する
- オプショナルチェイニング（`?.`）を活用する
- 型ガードを使用して安全に処理する

### 6. コンポーネント設計
- 単一責任の原則に従う（1つのコンポーネントは1つの役割）
- propsのインターフェースを明確に定義する
- 適切な粒度でコンポーネントを分割する
- 再利用可能な設計を心がける
